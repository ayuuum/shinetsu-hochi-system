## 2. ターゲットユーザーと利用シナリオ

### 2.1 ペルソナ定義

| ペルソナ | 役割 | 主な課題 | 利用頻度 |
|---|---|---|---|
| **A: 経営者・管理者** | iinuma氏、松岡氏 | 技術者の資格講習期限管理、現場配置判断 | 毎日 |
| **B: 総務・経理** | バックオフィス | 雇用保険、健康診断、車検、保険の期限管理 | 週3〜5回 |
| **C: 技術者 / 現場作業員** | 消防設備士等 | 自身の資格・講習履歴確認、証書アップロード | 月1〜2回 |

### 2.2 主要ユースケース

```mermaid
graph TD
    A[管理者] -->|UC-01| B(社員情報の一覧・検索)
    A -->|UC-02| C(資格講習期限アラート確認)
    A -->|UC-03| D(施工実績書の出力)
    A -->|UC-04| E(資格保持者一覧の抽出)
    
    F[総務・経理] -->|UC-05| G(健康診断・車検期限管理)
    F -->|UC-06| H(社員の入退社処理)
    
    J[技術者] -->|UC-07| K(自身の資格・講習履歴確認)
    J -->|UC-08| L(資格証書のアップロード)
    J -->|UC-09| M(アルコールチェック記録)

    A -->|UC-10| N(アルコールチェック一覧・未記録確認)
```

### 2.3 主要ユースケース詳細

#### UC-02: 資格講習期限アラートの確認
- **アクター**: 管理者、総務
- **基本フロー**:
  1. ダッシュボードに期限間近の講習・資格一覧が表示
  2. 「30日以内」「14日以内」「期限切れ」の3段階で色分け
  3. 該当社員の連絡先・資格詳細を確認
  4. 講習申込ステータスを記録
- **消防設備士特有のルール**:
  - 初回講習: 免状交付日以降の最初の4/1から**2年以内**
  - 2回目以降: 前回講習日以降の最初の4/1から**5年以内**
  - 免状写真更新: 交付後**10年以内**ごと

#### UC-03: 施工実績書の出力
- **アクター**: 管理者
- **基本フロー**:
  1. 対象社員を選択
  2. 出力対象期間を指定
  3. 施工実績（物件名、設備種別、施工内容、工期）をPDF/Excel出力

#### 3.3 自動アラート通知

##### アラートルール

| レベル | トリガー | 通知方法 | 色 |
|---|---|---|---|
| **INFO** | 期限60日前 | ダッシュボードのみ | 🔵 青 |
| **WARNING** | 期限30日前 | ダッシュボード + **メール** | 🟡 黄 |
| **URGENT** | 期限14日前 | ダッシュボード + **メール**（再通知） | 🟠 橙 |
| **CRITICAL** | 期限切れ | ダッシュボード + **日次メール** | 🔴 赤 |

##### 消防設備士の講習期限算出ロジック

```mermaid
flowchart TD
    A[消防設備士免状の取得日] --> B{講習受講履歴あり？}
    B -->|なし（初回）| C[免状交付日以降の<br/>最初の4/1を起算日]
    B -->|あり| D[最終講習日以降の<br/>最初の4/1を起算日]
    C --> E[起算日 + 2年 = 初回講習期限]
    D --> F[起算日 + 5年 = 次回講習期限]
    E --> G[アラート対象としてDB登録]

```

#### 3.10 アルコールチェック記録

> [!NOTE]
> 現状は Microsoft 365 の Excel シートを毎日開いて手入力しており、現場作業員の負担が大きい。
> スマホからワンタップで入力できる簡易フォームに置き換え、法令遵守の記録をシステムに一元化する。

##### 背景・法令要件
- 2023年12月1日施行の改正道路交通法により、白ナンバー事業者（一定台数以上保有）もアルコール検知器による確認が義務化
- 安全運転管理者による確認記録を**1年間保存**する義務あり

##### データ項目

| 検知器の値 | DECIMAL | | mg/L（任意入力） |
| 判定結果 | ENUM | ○ | 適正 / 不適正 |
| 備考 | TEXT | | |

##### 「不適正」（陽性）時のフロー（P1実装、仕様は確定）

```
作業員が「不適正」を選択
  → 確認ダイアログ「不適正として記録します。よろしいですか？」
  → 記録保存
  → 即時通知: 安全運転管理者にメール + ダッシュボード赤アラート
  → 画面表示: 「運転禁止 — 安全運転管理者の指示を仰いでください」
  → 再検査: 管理者が「再検査」ボタンで新規記録フォームを開く
  → 監査ログ: 全操作がタイムスタンプ付きで記録（編集・削除不可）
```

##### 管理者向け機能
- **一覧画面**: 日付・拠点・社員でフィルタ可能な記録一覧
- **未記録アラート**: 出勤しているのに記録がない社員をダッシュボードに警告表示
- **Excel出力**: 現行の365 Excelシートと同じフォーマットでのエクスポート（監査対応）
- **月次集計**: 拠点別・社員別の記録率レポート

## 6. データ取得・API設計

> [!NOTE]
> 本システムは **Server Component (SC) から Supabase を直接クエリ** するアーキテクチャ。
> 従来の REST API エンドポイントは不要。API Route は Cron ジョブとエクスポート機能のみ。

### 6.1 データ取得パターン

| 画面 | 取得方法 | 説明 |
|---|---|---|
| 社員一覧 | SC → `createSupabaseServer()` | フィルタは CC (Client Component) でローカル処理 |
| 社員詳細 | SC → `createSupabaseServer()` | 資格・施工実績を JOIN で取得 |
| 資格一覧 | SC → `createSupabaseServer()` | 社員・マスタを JOIN |
| 車両一覧 | SC → `createSupabaseServer()` | 直接クエリ |
| ダッシュボード | SC → `createSupabaseServer()` | 集計クエリ (count, group by) |

**ミューテーション:** Client Component 内で `supabase` (ブラウザクライアント) → INSERT/UPDATE/DELETE → `router.refresh()` で SC 再取得。

### 6.2 API Route（最小限）

| メソッド | パス | 説明 |
|---|---|---|
| GET | `/api/cron/daily-alert` | 日次アラートメール (Vercel Cron, CRON_SECRET で保護) |
| GET | `/api/employees/export` | 社員一覧CSV エクスポート（P1予定） |
| GET | `/api/employees/:id/career-export` | 施工実績書PDF出力（P1予定） |
| GET | `/api/alcohol-checks/export` | アルコールチェック記録 Excel エクスポート（P2予定） |

### 6.3 ダッシュボードデータ構造（SC内で取得）

```typescript
// employees/page.tsx (Server Component) の例
const supabase = await createSupabaseServer();
const { data } = await supabase
  .from("employees")
  .select("*, employee_qualifications(*, qualification_master(name))")
  .order("employee_number");
// → EmployeesClient (CC) に props で渡す
```
