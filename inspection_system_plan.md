## 1. 本書の目的

本書は、株式会社信越報知（以下「甲」）のコア業務である **消防用設備等の法定点検業務** をシステム化する「点検システム」モジュールについて、機能範囲・データ設計・画面構成・開発スケジュールの**計画案**を定義するものである。

本モジュールは既存の「社員・資格管理システム」（Next.js + Supabase）への**追加開発**として位置づけ、既存の社員台帳・資格管理・施工実績・アラート基盤を再利用する。

> [!NOTE]
> 当初 `detailed_prd.md` 3.9 で「点検スケジュール管理（年2回の義務点検）」として簡易構想されたが、
> 「顧客物件単位の詳細管理は後続検討」（確認事項 #5）とされ、簡易テーブル `inspection_schedules` は一度廃止された。
> 本書はその「後続検討」を正式な計画として具体化するものである。

## 2. 背景と業務理解

### 2.1 法令背景（消防法）

消防用設備等の点検は **消防法 第17条の3の3** により、防火対象物の関係者に義務づけられている。信越報知はこの点検業務を顧客から受託する立場にある。

| 区分 | 内容 | 周期 |
|---|---|---|
| **機器点検** | 外観・機能の確認（消火器、自火報、誘導灯 等） | **6ヶ月に1回** |
| **総合点検** | 設備を作動させての総合的な機能確認 | **1年に1回** |
| **点検結果報告** | 消防長または消防署長への報告 | 特定防火対象物：**1年に1回** / 非特定防火対象物：**3年に1回** |

> 実務上は「年2回点検」（機器点検2回、うち1回は総合点検を兼ねる）として運用されることが多い。

### 2.2 現状の課題（想定）

| # | 現状の課題 | 影響 |
|---|---|---|
| 1 | 顧客物件ごとの点検時期がExcel・紙・担当者の記憶で分散管理 | 点検期限の失念・受託案件の取りこぼし |
| 2 | 「次回いつ・誰が・どの物件を」点検するかが一覧化できていない | 月次の人員アサインが属人的 |
| 3 | 点検結果（不良箇所・是正状況）の蓄積がない | 過去履歴の参照に時間がかかる |
| 4 | 点検報告書の作成が手作業 | 1件あたりの事務コストが大きい |
| 5 | 物件と施工実績・担当技術者が紐づいていない | 情報が二重管理になる |

### 2.3 導入目的・達成指標

| 指標 | 現状（想定） | 目標値 |
|---|---|---|
| 点検期限の失念 | 年数件リスク | **年0件** |
| 「今月点検すべき物件」の把握時間 | 都度集計が必要 | **ダッシュボードで即時** |
| 点検結果の検索時間 | 5〜10分 | **10秒以内** |
| 点検報告書の作成時間 | 手作業 | **将来的にPDF出力で短縮** |

## 3. スコープと段階リリース方針

業務影響が大きいため、**3フェーズ**に分けて段階導入する。本計画では Phase 1 を最小実用範囲（MVP）とする。

| フェーズ | 範囲 | 位置づけ |
|---|---|---|
| **Phase 1（MVP）** | 顧客物件台帳 + 点検予定/実績の登録 + 期限アラート + 一覧/カレンダー | 本計画の主対象 |
| **Phase 2** | 点検結果明細（設備別・不良/是正）+ 点検報告書PDF出力 | 様式入手後 |
| **Phase 3** | 顧客ポータル / 報告書の電子提出 / モバイル現場入力（オフライン） | 運用定着後に判断 |

## 4. 利用者と権限

既存の3ロール（管理者 / 総務 / 技術者）を踏襲する。

| 機能 | 管理者 | 総務 | 技術者（本人） |
|---|---|---|---|
| 顧客物件台帳 | 登録/閲覧/更新/削除 | 登録/閲覧/更新 | 閲覧（担当物件のみ） |
| 点検予定（アサイン） | 登録/閲覧/更新/削除 | 登録/閲覧/更新 | 閲覧（自分の担当のみ） |
| 点検実績・結果入力 | 登録/閲覧/更新/削除 | 閲覧 | 登録（自分の担当のみ） |
| 点検報告書出力 | 出力 | 出力 | — |
| 点検マスタ（設備種別等） | 全機能 | — | — |

> RLS は既存の `get_user_role()` / `user_roles.employee_id` の仕組みを再利用し、技術者は自分がアサインされた点検のみ参照・入力可とする（`technician_rls` 方針に準拠）。

## 5. 機能要件

### 5.1 顧客物件台帳【REQ-INSP-001】

| 要件ID | 要件内容 |
|---|---|
| REQ-INSP-001-01 | 顧客物件（建物）の新規登録・編集・削除（論理削除）ができる |
| REQ-INSP-001-02 | 物件基本情報（物件名・顧客名・所在地・防火対象物区分・用途・延床面積）を管理する |
| REQ-INSP-001-03 | 防火対象物区分（特定/非特定）に応じた報告周期を保持する |
| REQ-INSP-001-04 | 物件ごとの担当技術者（主担当）を設定できる |
| REQ-INSP-001-05 | 物件に紐づく消防用設備等（消火器・自火報・誘導灯 等）を登録できる |
| REQ-INSP-001-06 | 物件ごとの点検契約情報（契約有無・点検周期・契約金額）を保持する |
| REQ-INSP-001-07 | 既存の施工実績（`construction_records`）と物件を関連づけられる |
| REQ-INSP-001-08 | フリーワード検索・拠点/区分フィルタ・CSVエクスポートができる |

### 5.2 点検予定・スケジュール管理【REQ-INSP-002】

| 要件ID | 要件内容 |
|---|---|
| REQ-INSP-002-01 | 物件ごとに次回点検予定日（機器点検・総合点検）を登録/自動算出できる |
| REQ-INSP-002-02 | 点検種別（機器点検 / 総合点検）を区別して管理する |
| REQ-INSP-002-03 | 前回点検完了日 + 周期 から次回予定日を自動算出する |
| REQ-INSP-002-04 | 点検予定に担当技術者をアサインできる（複数可） |
| REQ-INSP-002-05 | 月別・拠点別・担当者別に点検予定を一覧/カレンダー表示できる |
| REQ-INSP-002-06 | ステータス（未実施 / 実施済 / 延期 / 中止）を管理する |

### 5.3 点検実績・結果記録【REQ-INSP-003】

| 要件ID | 要件内容 |
|---|---|
| REQ-INSP-003-01 | 点検の実施日・実施者・点検種別・総合判定（良/要是正）を記録する |
| REQ-INSP-003-02 | （Phase 2）設備種別ごとの点検結果明細（良/不良・不良内容）を記録する |
| REQ-INSP-003-03 | （Phase 2）不良箇所の是正状況（是正済/未是正・是正予定日）を管理する |
| REQ-INSP-003-04 | 点検時の写真・帳票画像をアップロードできる（Supabase Storage） |
| REQ-INSP-003-05 | 点検完了時に物件の次回点検予定日を自動更新する |
| REQ-INSP-003-06 | （Phase 2）点検結果報告書をPDF出力できる |

### 5.4 点検アラート【REQ-INSP-004】

既存のアラート基盤（4段階：INFO/WARNING/URGENT/CRITICAL、画面内表示、メール通知なし）に統合する。

| 要件ID | 要件内容 |
|---|---|
| REQ-INSP-004-01 | 次回点検予定日が近い物件をダッシュボードにアラート表示する |
| REQ-INSP-004-02 | 点検期限切れ（予定日超過で未実施）を CRITICAL として表示する |
| REQ-INSP-004-03 | 報告期限（特定:1年/非特定:3年）が近い物件を警告する |
| REQ-INSP-004-04 | 「今月の予定」ウィジェット（REQ-001-04）に点検予定を連動表示する |

### 5.5 点検マスタ管理【REQ-INSP-005】

| 要件ID | 要件内容 |
|---|---|
| REQ-INSP-005-01 | 消防用設備等の種別マスタ（消火器/屋内消火栓/自動火災報知設備/誘導灯 等）を管理する |
| REQ-INSP-005-02 | 防火対象物の用途区分マスタ（消防法施行令 別表第一準拠）を保持する |

## 6. データモデル（Phase 1 + Phase 2）

既存テーブル（`employees` / `construction_records` / `alert` 系）を参照しつつ、点検系テーブルを追加する。

```
inspection_properties (顧客物件台帳)
  ├─ id, property_name, client_name, address
  ├─ building_use_category   -- 特定 / 非特定
  ├─ usage_detail            -- 用途（別表第一の項）
  ├─ total_floor_area
  ├─ inspection_cycle_months -- 点検周期（既定6）
  ├─ report_cycle_months     -- 報告周期（特定12 / 非特定36）
  ├─ primary_technician_id   -- FK → employees
  ├─ contract_flag, contract_amount
  ├─ next_device_inspection_date    -- 次回 機器点検予定
  ├─ next_comprehensive_inspection_date -- 次回 総合点検予定
  ├─ next_report_due_date           -- 次回 報告期限
  └─ deleted_at, deleted_by, timestamps

inspection_property_equipments (物件×設備)   ※物件に設置された消防用設備
  ├─ id, property_id (FK)
  ├─ equipment_type_id (FK → inspection_equipment_types)
  ├─ quantity, location_note
  └─ timestamps

inspection_visits (点検予定/実績)            ※1回の点検 = 1レコード
  ├─ id, property_id (FK)
  ├─ inspection_type         -- 機器点検 / 総合点検
  ├─ scheduled_date          -- 予定日
  ├─ completed_date          -- 実施日（NULL=未実施）
  ├─ status                  -- 未実施 / 実施済 / 延期 / 中止
  ├─ overall_result          -- 良 / 要是正（実施済のみ）
  ├─ notes
  └─ timestamps

inspection_visit_assignees (点検×担当者)     ※多対多アサイン
  ├─ visit_id (FK), employee_id (FK)
  └─ role  -- 主任 / 補助

inspection_results (点検結果明細／Phase 2)
  ├─ id, visit_id (FK)
  ├─ equipment_type_id (FK)
  ├─ result                  -- 良 / 不良
  ├─ defect_detail
  ├─ correction_status       -- 是正済 / 未是正 / 経過観察
  ├─ correction_due_date
  └─ timestamps

inspection_attachments (点検添付)
  ├─ id, visit_id (FK)
  ├─ storage_path, file_type, caption
  └─ timestamps

inspection_equipment_types (設備種別マスタ)
  ├─ id, name, category, sort_order, is_active
```

### 主な算出ロジック

- **次回機器点検予定** = 前回機器点検完了日 + `inspection_cycle_months`（既定6ヶ月）
- **次回総合点検予定** = 前回総合点検完了日 + 12ヶ月
- **次回報告期限** = 前回報告日 + `report_cycle_months`（特定12 / 非特定36ヶ月）
- 点検実績の登録時（REQ-INSP-003-05）に上記を再計算して物件レコードを更新

## 8. 既存システムとの統合方針

| 連携先 | 内容 |
|---|---|
| アラート基盤（REQ-004） | 点検期限・報告期限を既存4段階アラートに合流。ダッシュボード「今月の予定」に点検を表示 |
| 社員台帳（`employees`） | 担当技術者アサインで参照。技術者の `/today`・`/me` に自分の点検予定を表示 |
| 施工実績（`construction_records`） | 物件と施工実績を相互リンク。物件詳細から過去工事を参照 |
| 資格管理 | 点検は有資格者（消防設備点検資格者等）が実施。担当者の保有資格を表示（任意） |
| RLS / 権限 | 既存 `get_user_role()`・`technician_rls` を踏襲 |
| 性能ルール（TODOS.md） | 一覧は `PAGE_SIZE + 1`、未表示タブは初回 fetch しない、マスタは `unstable_cache` |

## 9. 非機能・制約

- 既存の NFR（HTTPS / RLS / 日次バックアップ / レスポンシブ / 性能目標）を踏襲する。
- 点検物件一覧は **0.8秒以内**、物件詳細は **1.0秒以内**（TODOS.md 目標秒数に追加）。
- 写真添付は Supabase Storage 無料枠（1GB）を考慮し、圧縮・上限枚数を検討する。
- メール通知は引き続き運用対象外（画面内アラートのみ）。
